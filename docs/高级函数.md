



在极简的实体（`Process` 和 `Log`）之上，我们可以构建出极其优雅且强大的高级功能。由于我们采用了“操作系统进程管理”的隐喻，很多高级功能都可以从 Linux 操作系统的核心理念中汲取灵感。

除了基础的增（`spawn`/`create`）、删（`kill`/`delete`）、改（`update`）、查（`ps`/`list`），我认为以下几个高级功能是让这个工具**产生质变**的关键：

### 1. 核心状态机与自动日志 (State Machine & Auto-Logging)
普通的 CRUD 只是改个字段，但在“进程管理”理念下，**状态的流转本身就是重要信息**。
*   **动作绑定日志**：当你改变一个进程的状态时，系统应该自动（或强制要求）插入一条 Log。
*   **具体场景**：
    *   `taskctl block <PID> -m "等待产品经理确认需求"`：将进程状态改为 `blocked`，同时自动生成一条内容为 `[System] Process blocked. Reason: 等待产品经理确认需求` 的 Log。
    *   `taskctl wake <PID> -m "需求已确认，继续开发"`：状态切回 `running`，同时记入 Log。
    *   `taskctl terminate <PID> -m "上线完成"`：标记任务完成。
*   **价值**：未来回顾时，你能清晰看到这个任务卡了几次、卡在哪里、怎么重新启动的，完美还原“思考过程”。

### 2. 上下文挂载 (Context Attach / Active Workspace)
这是极大提升 CLI 和 TUI 体验的杀手锏。
*   **痛点**：如果你正在专注处理 PID 42 的任务，每次写日志都要打 `taskctl log 42 "修复了UI错位"`，非常繁琐。
*   **高级功能**：引入“当前活跃进程（Active Process）”的概念（类似于 `git checkout` 或 `tmux attach`）。
    *   执行 `taskctl attach <PID>`：进入该进程的上下文。
    *   之后你只需要执行 `taskctl log "修复了UI错位"`，系统会自动把 Log 记在 PID 42 下。
    *   你的终端提示符（如果在 TUI 中）甚至可以显示当前正在 `[Process #42: 重构登录逻辑]` 中。

### 3. 进程树可视化 (Process Tree / `pstree`)
既然我们支持了 `parent_id`（无限子任务），就需要有强大的树状展示能力。
*   **高级功能**：类似 Linux 的 `pstree` 命令。
*   **具体场景**：执行 `taskctl tree` 或 `taskctl tree <PID>`。
*   **展示效果**：
    ```text
    [#1] 🚀 开发个人网站 (running)
    ├── [#5] ⏸️ 购买域名 (blocked: 等待备案审核)
    └── [#6] 🚀 开发前端界面 (running)
        ├── [#8] 🛑 设计 Logo (terminated)
        └──[#9] 🚀 编写响应式 CSS (running)
    ```
*   **价值**：一目了然地看到宏观目标是如何被拆解的，以及哪个底层子进程阻塞了整个父进程。

### 4. 全局时间轴与热力图 (Timeline & Activity Heatmap)
因为我们的 `Log` 是极其纯粹的时间流，我们可以完全脱离具体的 Process，以上帝视角审视时间。
*   **全局时间轴 (`taskctl timeline`)**：
    *   把今天（或本周）所有的 Log 按时间顺序拉出来。
    *   效果：“早上 9 点我在 [进程#5] 写了一段日志，10 点把 [进程#6] 挂起了，下午 2 点新建了 [进程#9]。” 这就是你完美的**每日工作日报（Daily Stand-up）**。
*   **活跃度统计 (`taskctl stats`)**：
    *   利用 SQLite 聚合查询，统计每天生成的 Log 数量和状态流转次数。
    *   在 TUI 或 Web 端渲染出类似 GitHub 的绿色热力图。

### 5. 沉淀与导出 (Knowledge Export / Dump)
一个好的任务管理工具，它的终点应该是**个人知识库（PKM）**。
*   **高级功能**：当你 `terminate` 掉一个长期进程（比如经历了一个月的项目），它不应该在数据库里吃灰。
*   **具体场景**：执行 `taskctl export <PID>`。
*   **价值**：将该进程的 Title、Description 以及所有的 Log（包含状态流转和你的笔记），按时间线组装成一个漂亮的 Markdown 文件。你可以直接把这个文件扔进 Obsidian 或 Notion 中，作为这个项目的**终极复盘文档**。

### 6. 模糊检索与过滤 (FTS & Querying)
配合 SQLite FTS5 的高级搜索。
*   **高级功能**：全局正文检索（不只是搜标题）。
*   **具体场景**：执行 `taskctl grep "数据库死锁"`。系统会遍历所有 Process 的描述和所有 Log 的内容，瞬间定位到你半年前记录的那条“遇到了数据库死锁，通过添加联合索引解决”的日志，并告诉你它属于哪个 PID。

回到你的核心问题：**在 `core` 核心逻辑层，除了最基本的 CRUD（增删改查）之外，这两个实体还需要哪些专属的操作函数？**

由于我们要严格遵守“核心库优先”的原则，所有涉及**业务逻辑**、**事务（Transaction）**和**复杂连表查询**的操作，都必须封装在 `core` 里。如果是让 AI 来写代码，绝不能让它在 CLI 或 Web 层去拼 SQL。

我梳理了一下，为了支撑起我们前面讨论的“进程管理”和“高级功能”，`core` 包里至少还需要定义以下 **5 类高级操作函数**：

### 1. 状态机与原子操作（State Machine）
这是最重要的！状态变更（比如挂起进程）不仅仅是更新 `Process.status`，还要**强制**写一条 `Log`。这两个动作必须在一个数据库事务（Transaction）里完成，要么全成功，要么全失败。

*   `ChangeProcessState(processID, newState, reasonLog string)`
    *   **底层逻辑**：开启事务 -> 更新 Process 的 status -> 在 Log 表中插入一条新记录（带有时间戳和变更原因） -> 提交事务。
    *   **价值**：保证数据绝对一致。不管你在 CLI、TUI 还是 Web 端操作，都不可能出现“状态变了但没记录原因”的脏数据。

### 2. 树状层级查询（Hierarchy & Tree）
为了支持无限子任务，并且为了以后 TUI 里能画出漂亮的 `pstree`（进程树），我们需要专门的递归查询函数。

*   `GetProcessTree(rootProcessID)`
    *   **底层逻辑**：可以通过 SQLite 的 `WITH RECURSIVE` 递归查询，或者在 Go 内存里组装。返回一个包含自身以及所有嵌套子进程的结构体（`ProcessNode`）。
*   `GetSubProcesses(parentID)`
    *   **底层逻辑**：只查直接子节点，常用于懒加载展开。

### 3. 全局时间轴与聚合（Timeline & Aggregation）
为了打破 Process 的孤岛，把所有的 Log 串成一条时间线（用于每日回顾或热力图）。

*   `GetGlobalTimeline(startTime, endTime)`
    *   **底层逻辑**：直接查 `Log` 表，根据 `created_at` 排序，并通过 `JOIN` 顺便带出它们所属的 `Process` 的标题（Title）。
    *   **价值**：用于生成“今天干了啥”的流水账日报。
*   `GetActivityStats(days)`
    *   **底层逻辑**：使用 `GROUP BY DATE(created_at)` 统计过去 N 天内每天产生的 Log 数量。
    *   **价值**：为前端生成 GitHub 风格的“提交热力图”提供直接数据。

### 4. 全文检索（Full-Text Search）
前面提到的跨表模糊搜索引擎。

*   `GlobalSearch(keyword string)`
    *   **底层逻辑**：利用 SQLite FTS5 虚拟表，同时在 `Process.title`、`Process.description` 和 `Log.content` 中匹配关键字。
    *   **返回值**：返回一个混合的搜索结果列表，告诉前端哪几个进程或哪几条日志命中了关键字。

### 5. 导出/组装上下文（Context Aggregation）
为了支持你说的单文件/多文件 Markdown 导出，需要一个极其强大的“打包提取”函数。

*   `GetProcessContext(processID)`
    *   **底层逻辑**：一并查出：该 Process 的全量信息 + 所有直接子进程列表 + 按照时间正序排列的所有 Log 记录。
    *   **价值**：把这个函数返回的完整数据结构交给一个模板引擎（Template），就能瞬间生成排版精美的 Markdown 文档。

关于你问的**“使用了 GORM 还需要写 SQL 吗？”**，这是一个非常关键的工程权衡问题。

**简短的回答是：90% 的情况不需要，但 10% 的高级特性（如全文检索、日期按天聚合聚合）使用原生 SQL（通过 GORM 的 `Raw` 方法）会更高效且优雅。**

具体来说：
*   **不需要写 SQL 的地方**：所有的增删改查（CRUD）、简单的条件过滤、关联查询（比如查出一个进程所有的 Log，GORM 用 `Preload` 一行代码搞定）、事务控制（`Transaction`）。这些 GORM 封装得非常完美。
*   **建议写一点原生 SQL 的地方**：
    1. **FTS5 全文检索**：因为虚拟表的 `MATCH` 语法是 SQLite 特有的，GORM 没有原生适配，用 `db.Raw("SELECT * FROM processes_fts WHERE processes_fts MATCH ?", keyword)` 最简单粗暴。
    2. **热力图的时间聚合**：像 `GROUP BY DATE(created_at)` 这种特定数据库的日期函数，用原生 SQL 写比用 GORM 的链式调用拼凑要直观得多。

GORM 最好的地方就在于**它不排斥原生 SQL**，你可以两者混用。

# Process Management - Core API Specification

## 1. Overview
The `core` package encapsulates all database interactions and business logic. It serves as the single source of truth for the CLI, TUI, and Web layers. We use `gorm` as the primary ORM, leveraging raw SQL via `gorm.Raw()` only for SQLite-specific features (like FTS5 and recursive CTEs).

## 2. Data Transfer Objects (DTOs)

To support advanced queries, we define several DTOs alongside our main entities (`Process` and `Log`).

```go
package core

import "time"

// ProcessNode represents a hierarchical process tree
type ProcessNode struct {
    Process
    Children[]*ProcessNode `gorm:"foreignKey:ParentID"` // Allows GORM Preload
}

// TimelineEntry represents a log entry joined with its parent process title
type TimelineEntry struct {
    Log
    ProcessTitle string `gorm:"column:process_title"`
}

// ActivityStat represents daily log counts for heatmaps
type ActivityStat struct {
    Date  string `gorm:"column:date"`  // Format: YYYY-MM-DD
    Count int    `gorm:"column:count"`
}
```
